# 产品与技术决策

## 名称

当前保留名称想法：`Seat Happens`

含义：

- 谐音幽默，和“抢不到座位”场景相关。
- 英文短，适合 TUM 国际环境。
- 可继续评估是否需要中文副标题。

## 技术栈

- 前端：单文件 HTML/CSS/JavaScript。
- 后端：Supabase。
- 部署：GitHub Pages。

选择原因：

- 用户规模初期 4-5 人，未来约 200 人，Supabase 足够。
- 不需要自己租服务器。
- GitHub Pages 免费、简单，适合静态前端。
- Supabase 提供 Auth、Postgres、Row Level Security。

## 账号方式

使用简单用户名、昵称、密码。

原因：

- 用户不需要绑定真实学校账号或 Google/GitHub。
- 降低隐私顾虑。
- 适合小范围测试。

注意：

- 不要在文档里记录用户密码。
- 管理员密码只由用户自己保存。

## 房间系统措辞

不要称为官方“预约”或“保证座位”。

推荐措辞：

- 房间共享
- 加入自习计划
- Captain 已到达
- 剩余座位
- 预计房型
- 到场确认

原因：

- StudiTUM 本身不提供此类官方软件或资源分配系统。
- 这个网页是同学之间协作共享，不代表官方占座。

## Captain 机制

Captain 是愿意早起到场、发布房间计划并确认房间的人。

设计目标：

- 鼓励用户承担早起成本。
- 给其他用户出发前的确定性。
- 用历史记录和称号建立信任。

积分暂时不是优先级。先做记录、称号、可靠性展示。

## 优先级

高优先级：

- 房间计划日历。
- Captain 创建和修改计划。
- 用户报名、取消、签到。
- 房间容量、剩余座位、主题、讨论模式。
- Captain 可信度。

中优先级：

- 新用户说明。
- 管理员课程/任务维护。
- 反馈。
- 聊天广场。

低优先级：

- 插座数量。
- 3D 地图。
- 课程任务库深化。
- 积分系统。
